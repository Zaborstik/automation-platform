# Руководство по внесению вклада (Contributing Guide)

> **Документ:** как участвовать в разработке (Issues, PR, стиль коммитов).  
> **К чему относится:** процесс разработки всего монорепозитория.

Спасибо за интерес к проекту automation-platform! Мы рады вашему участию.

> 📖 **Git workflow:** см. общий указатель [`INDEX.ru.md`](INDEX.ru.md), обзор [`README.ru.md`](README.ru.md) и шаблоны в каталоге **[`.github/`](../.github/)**.

## Как внести вклад

### Сообщение об ошибках (Bug Reports)

Если вы нашли ошибку, пожалуйста:
1. Проверьте, что проблема еще не описана в [Issues](https://github.com/Zaborstik/automation-platform/issues)
2. Создайте новый issue с шаблоном "Bug Report"
3. Укажите:
   - Описание проблемы
   - Шаги для воспроизведения
   - Ожидаемое и фактическое поведение
   - Версию Java, Maven и другие релевантные детали окружения

### Предложение новых функций (Feature Requests)

1. Проверьте, что функция еще не запрошена
2. Создайте issue с шаблоном "Feature Request"
3. Опишите:
   - Проблему, которую решает функция
   - Предлагаемое решение
   - Альтернативные варианты (если есть)

### Pull Requests

1. **Обновите локальную ветку main и создайте новую ветку:**
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/your-feature-name
   # или
   git checkout -b fix/your-bug-fix
   ```
   
   > 💡 **Важно:** Всегда обновляйте `main` перед созданием новой ветки!

2. **Сделайте изменения**:
   - Следуйте стилю кода проекта
   - Добавьте тесты для новой функциональности
   - Убедитесь, что все тесты проходят: `mvn test`
   - Проверьте качество кода: `mvn verify`

3. **Закоммитьте изменения**:
   ```bash
   git commit -m "feat: добавлена новая функция X"
   # или
   git commit -m "fix: исправлена ошибка Y"
   ```
   
   Используйте [Conventional Commits](https://www.conventionalcommits.org/):
   - `feat:` - новая функция
   - `fix:` - исправление ошибки
   - `docs:` - изменения в документации
   - `style:` - форматирование, отсутствующие точки с запятой и т.д.
   - `refactor:` - рефакторинг кода
   - `test:` - добавление тестов
   - `chore:` - обновление задач сборки, конфигурации и т.д.

4. **Отправьте изменения**:
   ```bash
   git push origin feature/your-feature-name
   ```

5. **Создайте Pull Request**:
   - Заполните шаблон PR
   - Убедитесь, что все CI проверки прошли успешно
   - Дождитесь code review

## Стандарты кода

- Используйте Java 21
- Следуйте Java Code Conventions
- Пишите понятные имена переменных и методов
- Добавляйте JavaDoc для публичных методов
- Покрывайте новый код тестами (минимум 70% покрытия)

## Процесс разработки

1. Все изменения проходят через Pull Request
2. Минимум один одобряющий review требуется для merge
3. Все CI проверки должны пройти успешно
4. PR должен быть актуальным (rebase на main при необходимости)

## Вопросы?

Если у вас есть вопросы, создайте issue с меткой `question` или начните Discussion.

Спасибо за ваш вклад! 🎉
