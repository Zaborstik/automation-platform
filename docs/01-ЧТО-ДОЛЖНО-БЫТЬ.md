# Целевая архитектура: Automation Platform

## Назначение системы

Automation Platform -- платформа для автоматизации действий в веб-приложениях. Система получает запрос пользователя (на естественном языке), парсит его через LLM, строит план выполнения из заранее известных действий и типов объектов, а затем выполняет этот план через браузерного агента (Playwright).

---

## Модули и их роли

### platform-core
Ядро системы. Содержит доменную модель (records), бизнес-логику планирования, машину состояний ЖЦ, интерфейс Resolver.
Не зависит ни от Spring, ни от БД -- чистая Java.

### platform-api
REST API (Spring Boot). Содержит JPA-сущности, репозитории, сервисы, контроллеры, Flyway-миграции. Является точкой входа для клиентов.

### platform-agent
Браузерный агент. HTTP-клиент к Playwright-серверу (Node.js). Преобразует шаги плана в команды браузера (клик, ввод текста, навигация и т.д.).

### platform-executor
Исполнитель планов. Берёт план, передаёт шаги в AgentService, собирает лог выполнения и агрегированный результат.

### platform-knowledge
Интеллектуальный слой. База знаний о целевых приложениях, LLM-клиент, парсер запросов, генератор планов, сканер страниц.

## Схема данных (целевая)

### Схема `system` -- справочники

| Таблица | Назначение |
|---------|------------|
| `workflow_step` | Шаги жизненного цикла: `new`, `in_progress`, `paused`, `completed`, `failed`, `cancelled` |
| `workflow` | Жизненный цикл (ЖЦ): `wf-plan`, `wf-plan-step`. Ссылка на первый шаг |
| `workflow_transition` | **[ЦЕЛЕВАЯ]** Допустимые переходы ЖЦ: `from_step` -> `to_step` для данного `workflow` |
| `action_type` | Типы действий: `navigation`, `interaction`, `data_input`, `validation`, `artifact` |
| `action` | Действия платформы: `open_page`, `click`, `input_text`, `select_option`, `wait_element`, `read_text`, `take_screenshot` |
| `entity_type` | Типы объектов: `page`, `form`, `input`, `button`, `link`, `table` |
| `action_applicable_entity_type` | Применимость действий к типам объектов (связка action <-> entity_type) |
| `ui_binding` | **[ЦЕЛЕВАЯ]** Привязка действия к UI-селектору (CSS/XPATH/TEXT) |
| `ui_binding_metadata` | **[ЦЕЛЕВАЯ]** Метаданные привязки к UI |

### Схема `zbrtstk` -- рабочие данные

| Таблица | Назначение |
|---------|------------|
| `plan` | План выполнения. Имеет ЖЦ (`workflow_step_internalname`), хранит шаг остановки (`stopped_at_plan_step`), цель (`target`), пояснение (`explanation`) |
| `plan_step` | Шаги плана. Каждый шаг -- мини-задача. Имеет свой ЖЦ, тип объекта (`entitytype`), объект действия (`entity_id`), порядок (`sortorder`) |
| `plan_step_action` | Действия шага. Связка `plan_step` <-> `action` + `meta_value` (например текст для ввода) |
| `plan_result` | Итог выполнения плана: `success`, время начала/окончания |
| `plan_step_log_entry` | Лог по шагам: сообщение, ошибка, время выполнения, вложение (скриншот) |
| `attachment` | Вложения (скриншоты ошибок и т.д.) |

## Два ключевых потока

### Поток 1: Первичная обработка (настройка среды)

Выполняется ДО начала работы с пользовательскими запросами.

```
1. Определяются action_type (типы действий):
   navigation, interaction, data_input, validation, artifact

2. Генерируются action (действия) на основе базы знаний
   и логики сканируемого приложения (RAD):
   open_page, click, input_text, select_option, wait_element, read_text, take_screenshot

3. Строится таблица применимости action_applicable_entity_type:
   open_page -> page
   click -> button, link
   input_text -> input
   select_option -> input
   wait_element -> page, form
   read_text -> table, page
   take_screenshot -> page

4. [В будущем] Сегментирование по entity_type
```

### Поток 2: Обработка пользовательского запроса

```
1. Пользователь отправляет запрос (естественный язык)
   -> LLM парсит запрос, уточняет детали
   -> Определяет entity_type, нужные action, параметры

2. Система находит "путь" к решению:
   Поскольку мы знаем, какие action применимы к конкретному entity_type,
   строится последовательность шагов

3. Создаётся plan:
   - Имеет свой ЖЦ (workflow: new -> in_progress -> completed/failed)
   - Хранит шаг ЖЦ (workflow_step_internalname)
   - Хранит шаг, на котором остановилось выполнение (stopped_at_plan_step)

4. Создаются plan_step (шаги плана):
   - Мини-задачи внутри плана
   - Каждый шаг имеет свой ЖЦ
   - Хранит тип объекта (entitytype) и сам объект (entity_id)
   - Имеет порядок (sortorder)

5. Создаются plan_step_action (действия шага):
   - У одного plan_step может быть несколько action
   - Хранит meta_value (например текст для ввода в поисковую строку)

6. План выполняется через браузерного агента:
   - Каждый шаг -> команда Playwright
   - Результат записывается в plan_result
   - При ошибке/прерывании:
     - Создаётся plan_step_log_entry
     - Создаётся скриншот (attachment)
```

## Целевые компоненты по модулям

### platform-core (должен содержать)
- Domain records: Plan, PlanStep, PlanStepAction, PlanResult, PlanStepLogEntry, Action, ActionType, EntityType, Workflow, WorkflowStep, WorkflowTransition, UIBinding, State, Attachment
- `Resolver` -- интерфейс поиска сущностей + переходов ЖЦ
- `InMemoryResolver` -- реализация для тестов
- `Planner` -- построение одношаговых и многошаговых планов
- `PlanPathFinder` -- поиск последовательности действий через граф применимости
- `PlanValidator` -- валидация целостности плана
- `LifecycleManager` -- машина состояний ЖЦ (валидация и выполнение переходов)
- `ExecutionEngine` -- координация Resolver + Planner
- `PlanUpdateRequest` -- DTO для обновления состояния плана

### platform-api (должен содержать)
- JPA-сущности для всех таблиц
- Репозитории для всех сущностей (включая PlanStep, PlanStepAction)
- CRUD-сервисы и контроллеры для Action, EntityType, Workflow
- PlanService с поддержкой листинга, поиска, переходов ЖЦ
- PlanExecutionService с обновлением статусов при выполнении
- DatabaseResolver с поддержкой переходов ЖЦ
- Flyway-миграции (V1-V4+)
- OpenAPI/Swagger документация

### platform-agent (должен содержать)
- AgentClient (HTTP к Playwright-серверу)
- AgentService с поддержкой всех команд: open_page, click, hover, type, wait, explain, select_option, read_text, take_screenshot
- RetryPolicy + логика ретраев при транзиентных ошибках
- StepExecutionCallback для отчётов о прогрессе
- AgentHealthCheck для проверки доступности сервера

### platform-executor (должен содержать)
- PlanExecutor (синхронный) с опцией stopOnFailure
- AsyncPlanExecutor (асинхронный, CompletableFuture)
- PlanExecutionResult, ExecutionLogEntry

### platform-knowledge (должен содержать)
- Доменная модель: AppKnowledge, PageKnowledge, UIElement, ParsedUserRequest
- KnowledgeRepository (in-memory)
- LLMClient (интерфейс + заглушка для тестов)
- PromptTemplate (загрузка и заполнение шаблонов промптов)
- UserRequestParser (парсинг запроса через LLM)
- PlanGenerator (генерация плана из ParsedUserRequest)
- AppScanner (сканирование HTML через JSoup)
- EntityTypeDiscovery (маппинг UI-элементов на entity_type)
- KnowledgeService (фасад: сканирование -> сохранение -> генерация)
