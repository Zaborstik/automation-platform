## Детали по каждому пробелу

### 1. LLM-интеграция (КРИТИЧЕСКИЙ)

**Что нужно:**
- Интерфейс `LLMClient` с методами `complete(prompt)`, `complete(systemPrompt, userMessage)`
- Заглушка `StubLLMClient` для тестов (возвращает предопределённые ответы)
- `UserRequestParser` -- парсит текст пользователя через LLM в структурированный запрос
- `PlanGenerator` -- из распарсенного запроса строит многошаговый план
- Шаблоны промптов (parse-user-request, generate-plan-steps, clarify-request)

**Почему критический:** Без этого система не может обрабатывать запросы пользователей на естественном языке -- ключевая функция описанной архитектуры.

### 2. База знаний (КРИТИЧЕСКИЙ)

**Что нужно:**
- Доменная модель: `AppKnowledge`, `PageKnowledge`, `UIElement`
- `KnowledgeRepository` (in-memory хранилище)
- `AppScanner` -- парсинг HTML (JSoup) для обнаружения UI-элементов
- `EntityTypeDiscovery` -- маппинг UI-элементов на entity_type
- `KnowledgeService` -- фасад: сканирование -> сохранение -> маппинг

**Почему критический:** Описанная логика предполагает генерацию action на основе базы знаний и логики сканируемого приложения. Без базы знаний этого не сделать.

### 3. Многошаговый планировщик (КРИТИЧЕСКИЙ)

**Что нужно:**
- Метод `Planner.createMultiStepPlan(List<ExecutionRequest>)` -- несколько шагов в одном плане
- `PlanPathFinder` -- поиск последовательности действий через граф применимости action_applicable_entity_type
- Валидация применимости каждого действия к типу объекта

**Почему критический:** Текущий Planner строит только 1 шаг. Для реального сценария (открыть страницу -> ввести текст -> нажать кнопку -> ждать результат) нужен многошаговый план.

### 4. Машина состояний ЖЦ (КРИТИЧЕСКИЙ)

**Что нужно:**
- `WorkflowTransition` record -- допустимые переходы (new -> in_progress, in_progress -> completed/failed, etc.)
- `LifecycleManager` -- валидация и выполнение переходов
- Расширение `Resolver` методами `findTransitions()`, `findTransition()`
- Flyway-миграция V4 для таблицы `workflow_transition`
- Реализация в `DatabaseResolver`

**Почему критический:** План хранит workflow_step_internalname, но никогда его не обновляет. stoppedAtPlanStep задаётся при создании и не меняется при выполнении. ЖЦ фактически не работает.
